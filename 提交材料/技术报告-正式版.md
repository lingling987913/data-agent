面向GNC设计要素审查的数据智能体技术报告

# **1 项目概述**

## **1.1 建设背景**

GNC分系统承担姿态确定、姿态控制、轨道控制、故障处置和任务模式管理等职责，是型号任务能力形成、在轨运行安全和工程风险控制的重要组成。GNC设计审查贯穿方案论证、设计报告定稿、关键节点评审、验证确认和质量问题归零等环节，审查对象具有材料类型多、信息密度高、跨材料关系复杂、专业判断责任重等特征。

传统GNC分系统评审通常需要集中姿态确定、姿态控制、FDIR、接口、验证等多个方向的专家，并在有限时间内完成数百页送审材料的审阅、问题识别和意见归并。多型号并行研制条件下，专家排期、材料准备、跨文档核查和证据归档成为制约评审效率和评审质量的重要因素。

本数据智能体研制目标并非建设单一文档问答工具，也并非仅对PDF进行解析展示，而是围绕GNC设计要素审查业务，形成可进入评审准备、专业审查、质量产保自查和问题闭环的智能辅助审查能力。

## **1.2 业务需求与建设必要性**

GNC设计要素审查的业务需求集中体现在审查效率、审查质量、证据链完整性、领域专业适配和复杂文档处理五个方面。数据智能体应将专家从重复性阅读、材料整理、条目匹配和证据定位工作中释放出来，使专家重点投入方案合理性、风险接受性和型号放行等高价值判断。

# **2 GNC设计要素审查业务分析**

## **2.1 送审材料构成**

GNC设计审查通常面向送审包开展综合核查。送审包至少应包括任务输入、被审设计报告、接口或验证材料、检查单或质量自查材料。材料齐套程度直接决定审查能否进入深度核查。

表2-1 送审材料构成与缺失影响

| **送审包角色** | **典型材料** | **缺失影响** |
| --- | --- | --- |
| 任务依据 | 任务书、研制要求、载荷任务要求。 | 无法判断设计是否承接任务要求。 |
| 被审对象 | GNC设计报告、姿态确定/姿态控制设计说明。 | 无法开展设计要素核查。 |
| 接口依据 | ICD、遥测遥控定义、总线通信约束。 | 接口一致性和跨专业协同无法核查。 |
| 验证依据 | 仿真报告、试验报告、验证计划和结果表。 | 验证充分性和设计闭合无法判定。 |
| 检查单与质量材料 | 技术方案检查单、质量产保自查表、评审问题清单。 | 检查单响应和问题闭环无法核查。 |
| 历史参照 | 历史问题、典型缺陷、归零记录。 | 同类问题复用不足，经验传承不充分。 |

## **2.2 审查流程**

GNC设计要素审查可按“接收—结构化—门控—提取—匹配—核查—合稿—审定—确认—闭环”组织。数据智能体在该流程中承担材料预处理、缺项检查、设计要素抽取、证据定位和问题草稿生成职责；专家承担专业裁定、风险接受和正式结论确认职责。

## **2.3 人工审查痛点**

GNC设计审查的现实痛点不只体现为“材料多”，更体现为专家资源稀缺、专业判断难统一、跨文档一致性难核查、验证充分性难量化和历史知识难复用。上述痛点共同决定了数据智能体必须围绕业务流程建设，而不能停留在通用问答或简单OCR层面。

表2-3 人工审查痛点与业务影响

| **痛点** | **典型表现** | **业务影响** |
| --- | --- | --- |
| 审查时间长、专家资源稀缺 | 一次GNC评审往往需要多方向专家集中审阅数百页材料，多型号并行时排期冲突突出。 | 关键节点评审准备周期长，专家大量时间消耗在重复阅读和材料比对上。 |
| 审查深度依赖个人经验 | 新晋专家可能遗漏隐蔽问题，不同专家判断口径存在差异，长时间评审后漏检风险增加。 | 审查质量不稳定，问题发现依赖个人经验积累。 |
| 跨文档一致性核查不足 | 设计报告、ICD、仿真报告和试验报告由不同团队维护，参数和版本容易不同步。 | 接口类问题和参数不一致问题可能延后暴露，归零成本增加。 |
| 验证充分性难以量化 | 设计指标与验证条目缺少显式映射，极限、故障和组合工况覆盖盲区难系统识别。 | 无法充分回答“哪些设计要求已验证、哪些工况未覆盖”。 |
| 专家经验断层和知识孤岛 | 关键设计准则、历史教训和边界工况识别经验分散在个人经验和型号材料中。 | 新晋工程师学习周期长，历史问题跨型号复用不足。 |

## **2.4 传统AI方案适用边界与技术挑战**

传统AI方案直接用于GNC设计审查存在明显边界。审查工作要求专业判断可信、上下文关系完整、多专业意见可协同、每条意见有依据可查。仅依靠通用大模型问答或切块检索，难以满足航天评审对专业性、可追溯性和责任边界的要求。

## **2.5 证据链与审查合法性要求**

航天工程评审强调“意见必须有依据”。无法说明来源页码、原文片段、规范条款或历史问题参照的意见，不宜作为正式审查发现进入问题归零流程。数据智能体应将证据组织作为审查主线，而非在生成结论后再补充出处。

## **5.6 文档解析与结构化底稿构建**

在 GNC 设计要素审查的复杂业务场景中，送审包通常包含多份、多格式、版面复杂文件组，涉及的文件类型包括Word、Excel、PDF、图片、扫描件等。因此，文档解析与结构化底稿构建是形成审查能力的基础环节。原始送审材料不能直接支撑审查，必须转换为章节清晰、证据可定位、要素可抽取、质量可标记的结构化底稿。底稿内容包括章节结构、段落块、表格块、公式块、图片块、参数候选、设计要素候选和证据集合。

针对航天GNC相关的文档材料，我们在数据处理的过程中面临诸多主要难点：

一是复杂版面难以稳定还原，尤其横向扫描 PDF 导致参数表行序错乱、跨页表断裂和表格结构异常；

二是手写体字段在 OCR 后易出现“形似实错”（如手写符号、关键产品序号等），若未经校准即进入审批流程，会产生虚假的“不符合”结论；

三是插图与扫描图片难以获得可检索、可引用的文字依据，框图与曲线往往只能依赖人工看图，无法与章节、检查项在同一套依据池中对照。传统“全文抽取”或单一 OCR 无法同时解决上述问题，且解析失败时若不能显式降级与留痕，会拖慢整包审查并损害评审结论的可追溯性。错误示例如下图所示。

### **5.6.1 PDF类材料解析处理**

PDF类材料通常承载任务书、设计报告、接口文件和仿真报告，是GNC审查的主要材料类型。PDF解析处理应优先保留页码、章节、段落、表格、公式、图片和图注的版面关系。对于文字型PDF，系统进行正文抽取、章节恢复和表格识别；对于扫描PDF，系统采用OCR或外部解析能力进行图文识别；对于复杂排版PDF，系统记录低可信区域和解析降级状态。

PDF解析采用多级降级策略。高精度版面解析可用时，优先形成包含版面块的结构化对象；高精度解析失败时，切换至本地解析或文本抽取；扫描质量不足或工程图难以解析时，输出人工复核标识。

这部分是核心难点，也在后续文档解析创新处理里做详细的阐述。

### **5.6.2 Word类材料解析处理**

Word类材料通常包含设计说明、检查单、评审意见和整改记录。系统应保留标题层级、段落顺序、表格结构、批注和编号关系。对Word材料的解析重点在于恢复章节层级、抽取检查项、识别表格参数和保持原文段落可追溯。

Word材料中的表格、编号条款和批注信息是审查依据的重要来源。系统将条款编号、章节路径和表格单元格关联到证据集合，供后续规则匹配、条款响应检查和问题闭环使用。

该部分主要对Word文档中的图片增加VLM处理能力，与PDF材料中的图片处理逻辑保持一致。

### **5.6.3 Excel和表单类材料解析处理**

Excel和表单类材料主要包括检查单、自查表、问题清单和整改闭环表。系统按工作表、行列、单元格、表头和合并单元格关系进行结构化，重点识别检查条目、响应状态、责任项、问题编号、整改措施、复查结论和关闭状态。

该类材料解析的核心是保持表格语义，而不是仅抽取文本。系统需将表头与数据行建立关系，将检查项与响应内容、证据说明和闭环状态对应起来，为文文一致性审查和问题闭环提供结构化输入。

### **5.6.4 图片、扫描件与公式图表处理**

图片、扫描件和公式图表的处理以区域定位、OCR识别、低可信标记和人工复核为基本原则。系统对公式图片、趋势图、仿真曲线和工程图进行位置记录和说明性摘录，不将复杂工程图自动判读结果作为高置信结论依据。

对于公式和图表，系统应记录原文位置、图表编号、图注、关联章节和解析可信度。涉及关键公式、控制参数、仿真曲线和工程图判读的内容，应在审查结果中保留人工确认标识。

### **5.6.5 章节结构、证据集合与底稿对象构建**

不同类型材料解析结果最终统一到结构化底稿对象中。系统将章节结构、段落块、表格块、公式块、图片块和证据摘录统一编码，并建立材料来源、页码、章节路径和原文片段之间的映射关系。结构化底稿同时保存解析质量状态，用于后续审查路线判断和人工复核。

### **5.6.6 文档解析处理创新点**

本环节的设计目标，是将复杂送审材料转化为来源可定位、字段可校验、过程可追溯的结构化材料包，为此，本方案围绕三类痛点，在MinerU原有能力上进行算法集成：

1. 阅读方向评分与扫描件方向归一化：本方法对PDF或图片逐页判断：对宽高比横版且文本稀疏的扫描页，通过行纹理、边缘对比和版心分布计算可读性得分，在90°和270°中选择最佳旋转角度；竖版页则判断0°/180°翻转，形成“先转正、再识别、再修表”的可信处理链。解决因横向打印再扫描等导致的页面朝向与阅读方向不一致问题，避免OCR和版面分析出现表格行列错位、跨页断裂等异常；

表6-4 GNC设计文档审查验证输入

表6-5 GNC设计文档审查验证输出

| --- | --- |

| 设计要素清单 | 姿态确定、姿态控制、执行机构、传感器、控制律、参数、验证项等候选对象 |
| 专业审查发现 | 重大问题、主要问题、一般问题、建议项和待专家确认项 |

## **6.3 场景四：文件组文文一致性审查验证**

验证任务书、设计方案、接口控制文件、仿真验证报告、技术方案检查单等文件组之间的一致性审查能力。该场景重点回答任务要求是否被设计承接、设计指标是否被验证支撑、检查单条款是否有对应依据、接口参数是否前后一致。

表6-6 文件组文文一致性审查材料角色

| **材料角色** | **审查作用** |
| --- | --- |
| 任务书/研制要求 | 作为需求源头，提供任务指标、可靠性要求、安全要求和验证约束 |
| GNC设计方案/设计报告 | 作为被审对象，说明方案配置、算法模型、参数取值和接口设计 |
| 接口控制文件 | 用于核查信号定义、时序、数据刷新率、总线约束和接口一致性 |
| 仿真验证报告/试验报告 | 用于核查设计指标、边界工况、故障工况和组合工况覆盖情况 |
| 技术方案检查单 | 用于核查条款响应、证据充分性和审查准入条件 |

数据智能体先进行材料角色识别和文件组齐套性判定，再以结构化底稿为基础构建需求—设计—验证候选追溯矩阵。对显式编号、指标名称、单位、工况、接口参数和检查单条款进行跨文档比对，形成一致、部分一致、不一致、缺少依据和待确认等状态。

表6-7 文件组文文一致性审查输出

| **输出项** | **说明** |
| --- | --- |
| 文件组齐套性结果 | 判断任务书、设计报告、检查单、接口文件、验证材料是否满足审查准入 |
| 需求—设计—验证追溯矩阵 | 建立任务要求、设计响应、验证条目之间的候选关联 |
| 指标一致性清单 | 发现指标名称、数值、单位、工况或版本基线不一致问题 |
| 检查单响应状态表 | 形成已响应、部分响应、未响应和待专家确认状态 |
| 跨文档问题建议 | 识别设计报告与ICD、仿真报告与设计参数、检查单与证据之间的不一致 |

该场景验证数据智能体能够对多份并行演进的型号材料进行结构化比对，减少人工交叉翻阅和遗漏。对指标口径差异、版本基线差异和隐含工程耦合关系，数据智能体提供证据定位和待确认提示，由专家作最终判断。

## **6.5 综合验证结论**

五类场景覆盖了数据智能体从材料解析、专业审查、文件组一致性、通用化任务到质量闭环的主要能力边界。验证结果表明，数据智能体已具备以MinerU解析为基础的复杂文档结构化能力，并能够在结构化底稿上组织GNC审查、文文一致性审查和通用智能审查。当前能力重点适用于评审前辅助审查、设计报告复查、材料齐套性预检、检查单响应核查和质量问题闭环整理；正式审查结论仍须由专家复核后形成。

表6-10 综合验证结论与深化方向

| **能力维度** | **当前验证结论** | **后续深化方向** |
| --- | --- | --- |
| 材料解析 | MinerU已作为复杂文档解析主通道，pdftotext等作为降级兜底 | 提升低质量扫描件、复杂表格、公式语义和工程图表理解能力 |
| GNC审查 | 可形成设计要素候选、问题建议和证据定位 | 完善专业规则库、单元级深审和总师审定策略 |
| 文文一致性 | 可支持任务书、设计报告、检查单、接口和验证材料的追溯候选构建 | 增强隐含逻辑关系和版本基线差异判断 |
| 通用智能审查 | 可对非标准任务进行结构化、初筛和报告草稿生成 | 扩展更多专业领域装配和任务模板 |

后续建设应重点完善五类能力：一是进一步强化MinerU精准解析与人工复核闭环，提高图片式PDF、复杂表格、公式和图注的可用性；二是完善GNC专业规则库和历史问题库，将专家经验转化为可重复执行的审查规则；三是增强文文一致性审查，提升任务书、方案、接口、仿真和检查单之间的追溯覆盖；四是完善质量评测和基准回归，形成可量化、可复现的验收依据；五是对接型号数据中心、知识库、流程系统和质量管理系统，形成可持续运行的审查能力平台。

# **7 应用前景**

面向GNC设计要素审查的数据智能体具备沿研制流程纵向深化、向其他控制专业横向推广、向型号审查知识资产沉淀演进的应用基础。其核心价值不局限于单次评审提效，而在于将材料结构化、审查规则、历史问题、证据链和专家确认过程沉淀为可复用能力。

## **7.1 全生命周期AI审查能力深化**

在需求分析阶段，数据智能体可围绕需求文档中的性能、可靠性、安全性和接口指标，建立需求到设计要素的映射关系，辅助识别需求分解不完整、指标口径不一致和需求可实现性不足等问题。在方案设计阶段，可继续深化当前GNC审查能力，对设计输入、方案配置、控制指标、参数表、仿真验证和质量检查单进行关联审查，支撑方案完整性、一致性和可验证性判断。

在软件开发阶段，可将需求和设计要素映射到软件架构、接口定义和代码实现，通过静态分析、接口检查和规范审查识别功能遗漏、过度设计、时序不一致和资源占用异常等问题。在集成测试阶段，可构建设计指标与测试用例之间的映射关系，识别关键设计要素是否被充分验证，并结合实测数据分析设计预期与测试结果之间的偏差。在在轨服务阶段，可进一步将遥测数据与设计基线关联，对性能偏离、冗余设计有效性和容错机制实际效果进行分析，为后续型号迭代提供依据。

## **7.2 向其他控制专业推广**

卫星GNC、导弹制导控制、运载火箭姿控与导航控制等领域均具有复杂控制理论、高可靠性设计、多文档协同和严格评审闭环等共性。数据智能体在GNC领域形成的材料解析、结构化底稿、审查流程编排、追溯矩阵、专业规则库和问题闭环机制，可经领域适配后迁移至其他控制专业。

推广时应保持通用底座稳定复用，重点替换领域术语、审查规则、检查单条款、专业知识库、历史问题库和输出模板。可直接迁移的能力包括需求解析与验证、架构设计审查、接口一致性核查、验证充分性检查、质量问题闭环和审查证据链组织。需要补充建设的内容包括导弹、火箭等领域专用指标体系、典型失效模式、控制律审查规则、飞行剖面约束和专业验证工况库。

## **7.3 向型号审查知识资产演进**

数据智能体应从单次辅助审查工具逐步演进为型号审查知识工程底座。第一阶段形成结构化材料底稿库，沉淀任务书、设计报告、接口文件、仿真报告和检查单的章节结构、参数表、公式候选和证据集合。第二阶段形成专业规则库，将检查单条款、模板要求、典型问题、参数单位和验证覆盖规则结构化。第三阶段形成历史问题复用库，将评审发现、整改措施、复查结论和关闭依据按型号、阶段、专业和问题类型归档。第四阶段形成跨型号审查知识资产，支撑跨型号问题复用、质量趋势分析和专家经验传承。

# **8 人工复核要求**

数据智能体可自动完成材料接收、格式识别、文档结构化、材料齐套性检查、章节缺项提示、检查单候选匹配、设计要素候选提取、显式跨材料一致性提示、问题建议生成、意见单草稿和过程留痕。

以下内容必须人工确认：材料角色歧义、关键参数抽取、指标不一致原因、检查单部分响应、隐含追溯关系、重大问题严重度、整改建议、正式审查结论、风险接受和放行建议。系统输出属于智能辅助审查结果，不能替代专家审查责任、质量确认责任和型号决策责任。

# **9 结论**

面向GNC设计要素审查的数据智能体已形成以Super Agent为统一入口、以GNC设计审查、文文一致性审查和通用智能审查为三类能力路径、以结构化底稿、追溯矩阵、动态专家组队和质量评测为共用支撑机制的技术方案。该方案将多格式材料解析、任务规划、工具调用、专业核查、证据追溯、问题合稿和过程留痕统一到可运行链路中，能够支撑评审前辅助审查、设计报告复查、检查单执行、质量产保自查和问题闭环管理。

后续应重点完善复杂扫描件和工程图解析、模板章节深度门控、关键参数和公式校核、质量闭环状态表、GNC专业规则库、历史问题库、真实型号基准样例和生产级压测机制，逐步由单场景辅助审查工具演进为型号审查知识工程底座。
