## 2. 评审关注点与系统对应关系

评审通常要求日志或结果中能追溯：**任务输入 → 执行步骤 → 工具/解析器调用 → 最终输出**。本系统无单一「全链路审计日志文件」，字段分布在 API 轮询响应、工作流事件流、DAG/评测 Trace 与 `storage/runs/` 持久化 JSON 中，对应关系如下。

| 评审关注字段 | 主要承载位置 | 说明 |
| --- | --- | --- |
| **任务输入** | 竞赛 `POST /task/submit` 请求体；`DocumentTask.documents` / `task_description`；Review-Plus 上传材料；Super Agent `CreateSuperAgentRunRequest` | 竞赛任务完成后快照见 `storage/runs/tasks/{task_id}.json` 顶层字段 |
| **执行步骤** | 竞赛 `GET /task/status` 的 `current_step`、`progress`；Review-Plus `GET .../events`；GNC/Super Agent run 的 `current_step`、向导 `wizard_step` | 步骤名为字符串（如 `ingest_documents`、`parse:{file_name}`、`dag_pipeline`） |
| **工具/解析器调用** | `parser_trace`、`parser_fallback_logs`；规划 DAG `ExecutionTrace.parser_fallback_logs`；`RunTrace.parser_trace_summary`；Super Agent `skill_traces`（`tool_name`） | 解析降级由 `data_agent/parsing/fallback_trace.record_parser_fallback` 写入 |
| **最终输出** | 竞赛 `GET /task/result`；Review-Plus `findings` / `report`；Super Agent run JSON；离线基准 stdout/`--json-out` | 结构化结果模型见 `StructuredTaskResult`（`data_agent/domain/material_roles.py`） |

## 4. 日志与结果记录的位置与格式

### 4.1 应用与 Agent 文本日志

| 类型 | 位置 / 方式 | 格式 |
| --- | --- | --- |
| FastAPI / 业务 | uvicorn **stdout** | 文本行；级别 `LOG_LEVEL`（默认 INFO） |
| Agno Agent | `AGNO_LOG_PATH`（默认 `.dev/logs/agno.log`） | 文本 |
| 开发调试埋点 | `AGENT_DEBUG_LOG_PATH`（默认 `.dev/logs/agent-debug.log`） | NDJSON 风格调试事件 |

上述日志面向运维排障，**不保证**包含完整业务结构化字段；评审追溯建议以 §4.2–4.4 为准。

### 4.2 竞赛任务 API（推荐评测脚本路径）

执行逻辑：`data_agent/services/document_task_service.run_document_task` → 完成后 `DocumentTaskQueue._persist_trace` 写入 `storage/runs/tasks/`。

**轮询记录（过程）** — `GET /api/v1/task/status/{task_id}`：

```json
{
  "task_id": "...",
  "status": "running",
  "progress": 0.45,
  "current_step": "document_parsing",
  "scenario": "package_review",
  "parser_trace": [],
  "error": null,
  "created_at": "...",
  "started_at": "...",
  "completed_at": null
}
```

**结果记录（终态）** — `GET /api/v1/task/result/{task_id}`（未完成返回 409）：

- 顶层：`status`、`result`（`StructuredTaskResult`）
- `result.parser_trace`：解析器链路与降级
- `result.structured_output.parser_trace_summary`、`execution_metrics_snapshot`：质量快照
- `result.findings`、`review_report_markdown` 等：审查输出

字段定义见 [API接口说明.md](./API接口说明.md) 与 `data_agent/api/schemas.py`（`TaskStatusResponse`）。

### 4.3 Review-Plus（11 步）与 GNC（10 步）

| 链路 | 步骤数 | 步骤记录方式 | 查询接口 |
| --- | ---: | --- | --- |
| Review-Plus | 11 | Agno `Step` 名 + `record_event` 事件流（`sequence`、`type`、`payload`、`created_at`） | `GET /api/v1/review-plus/reviews/{id}/events`、`.../agent-traces` |
| GNC workflow | 10 | `current_step` 与步骤 JSON 产物 | `GET /api/v1/gnc-review/...`（见 OpenAPI **gnc-review** 标签） |

Review-Plus 11 步名称（工作流注册顺序）：`material_classification` → `scenario_detection` → `document_parsing` → `document_structuring` → `chief_orchestration` → `rule_extraction` → `rule_section_mapping` → `item_review` → `traceability` → `cross_document_review` → `report_composition`（`data_agent/workflows/review_plus_workflow.py`）。

GNC 十步：`review_intake` … `review_closure`（`data_agent/integrations/satellite_review/gnc_workflow.py`）。

### 4.4 规划 DAG 与 Super Agent

| 能力 | 持久化 / API | 关键结构 |
| --- | --- | --- |
| 规划执行 | `storage/runs/traces/{plan_id}.json`；`GET /api/v1/planning/trace/{plan_id}` | `RunTrace.execution_plan`（`ExecutionTrace`：`dag.nodes`、`completed_nodes`、`parser_fallback_logs`、`node_outputs`） |
| Super Agent | `storage/runs/review_data_agent_runs/{run_id}.json`；`GET .../runs/{id}/traces` | `skill_traces[]`（`skill_id`、`tool_name`、`input_summary`、`output_summary`）；`trace_report` |

解析降级单条格式（`parser_fallback_logs` / `record_parser_fallback`）：

```json
{
  "timestamp": "2026-05-31T12:00:00+00:00",
  "source_parser": "mineru_local",
  "fallback_parser": "pdftotext",
  "reason": "...",
  "recovered": true,
  "elapsed_ms": 120
}
```

### 4.5 离线基准（任务三）

命令：

```bash
python3 benchmark/run_golden.py \
  --json-out 提交材料/测试结果/基准运行摘要.json \
  --markdown-out 提交材料/测试结果/基准摘要.md \
  --fail-on-gate
```

输出 JSON 顶层字段：`benchmark_id`、`case_count`、`passed_count`、`pass_rate`、`average_structure_tree_f1`、`p95_elapsed_ms`、`gate_passed`、`results[]`（每 case 含 `case_id`、`elapsed_ms`、`evaluation_metrics`、`parser_trace` 相关指标、`errors` 等）。

**说明**：`golden_manifest.json` 中 `gnc_single_doc`、`gnc_multi_doc`、`review_plus_three_slot` 引用 `tests/fixtures/review_docs/*.md`；若 `tests/` 未检出，这些 case 会降级失败，门禁可能不通过（见 §7）。

## 5. 如何执行测试任务并获取完整可追溯记录

### 5.1 评审机最小环境

```bash
cd /path/to/data-agent
pip install -e ".[dev]"
cp .env.example .env
# 建议评审机关闭 MinerU，与 部署与运行说明.md §2.5 一致：
# MINERU_LOCAL_ENABLED=0
```

### 5.2 路径 A — 竞赛异步任务（最接近「一键评测」）

```bash
export TOKEN=dev-token-change-me
export BASE=http://127.0.0.1:8080
uvicorn data_agent.main:app --host 0.0.0.0 --port 8080

FIX="$(pwd)/提交材料/示例/04-规划与编排/测试数据"
TASK_ID=$(curl -s -X POST "$BASE/api/v1/task/submit" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d "{\"task_description\":\"月兔一号飞轮文档包审查\",\"documents\":[
    {\"file_name\":\"月兔一号_产品保证检查单.docx\",\"content_type\":\"path\",\"content\":\"$FIX/月兔一号_产品保证检查单.docx\"},
    {\"file_name\":\"月兔一号_飞轮研制任务书.docx\",\"content_type\":\"path\",\"content\":\"$FIX/月兔一号_飞轮研制任务书.docx\"},
    {\"file_name\":\"月兔一号_飞轮设计分析报告.docx\",\"content_type\":\"path\",\"content\":\"$FIX/月兔一号_飞轮设计分析报告.docx\"},
    {\"file_name\":\"月兔一号_文档检查需求.xlsx\",\"content_type\":\"path\",\"content\":\"$FIX/月兔一号_文档检查需求.xlsx\"}
  ]}" | python3 -c "import sys,json; print(json.load(sys.stdin)['data']['task_id'])")

# 过程日志：保存每次 status 轮询
while true; do
  curl -s "$BASE/api/v1/task/status/$TASK_ID" -H "Authorization: Bearer $TOKEN" \
    | tee -a 提交材料/测试结果/task_${TASK_ID}_status.log.jsonl
  STATUS=$(python3 -c "import json; print(json.load(open('/dev/stdin'))['data']['status'])" < 提交材料/测试结果/task_${TASK_ID}_status.log.jsonl 2>/dev/null || echo running)
  # 建议用 jq 判断；此处简化为轮询若干次后取 result
  sleep 2
  curl -s "$BASE/api/v1/task/status/$TASK_ID" -H "Authorization: Bearer $TOKEN" \
    | python3 -c "import sys,json; d=json.load(sys.stdin)['data']; import sys; sys.exit(0 if d['status'] in ('completed','failed') else 1)" && break
done

curl -s "$BASE/api/v1/task/result/$TASK_ID" -H "Authorization: Bearer $TOKEN" \
  | tee 提交材料/测试结果/task_${TASK_ID}_result.json

# 磁盘持久化（服务进程内已完成，可再次核对）
ls -la storage/runs/tasks/${TASK_ID}.json
```

可追溯组合：**submit 请求体** + **status 轮询序列** + **result JSON** + **`storage/runs/tasks/{id}.json`**。

### 5.3 路径 B — Review-Plus 分步（步骤 + 事件流）

1. `POST /api/v1/review-plus/reviews` 创建任务  
2. `POST .../upload` 上传材料  
3. `POST .../start` 启动 11 步 workflow  
4. 轮询 `GET .../events` 保存事件数组；可选 `GET .../agent-traces`  
5. 终态：`GET .../findings`、`.../report.md`  

任务体持久化：`storage/runs/review_plus_tasks/{review_id}.json`（含完整 `events`）。

### 5.4 路径 C — Super Agent（任务一跨文档，示例 03）

见 [示例/03-跨文档指代/README.md](./示例/03-跨文档指代/README.md) 与 [部署与运行说明.md](./部署与运行说明.md) §4.5：`POST .../super-agent/runs` → `POST .../runs/{id}/execute`（multipart 文件）。

追溯：`storage/runs/review_data_agent_runs/{run_id}.json` 与 `GET .../runs/{id}/traces`。

### 5.5 路径 D — 离线基准与 pytest（任务三 / 回归）

```bash
# 离线基准（生成 提交材料/测试结果/ 下的 JSON/Markdown）
python3 benchmark/run_golden.py \
  --json-out 提交材料/测试结果/基准运行摘要.json \
  --markdown-out 提交材料/测试结果/基准摘要.md

# pytest（需 tests/ 目录存在）
pytest -q tests/test_golden_benchmark.py tests/test_pdf_auto_parse_chain.py \
  tests/test_review_plus_workflow.py tests/test_gnc_workflow.py --tb=no \
  2>&1 | tee 提交材料/测试结果/单元测试摘要.txt
```

## 7. 已知限制与后续改进

| 限制 | 现状 | 建议 |
| --- | --- | --- |
| **无统一审计日志文件** | 字段分散在 API、JSON 持久化、stdout | 评审按 §5 组合抓取；后续可增「导出 trace 包」CLI |
| **`tests/` 可能未随提交包检出** | `pyproject.toml` 与 `golden_manifest` 仍引用 `tests/fixtures/` | 完整 clone 含测试树；或仅用 §5.1–5.2 竞赛 API + 示例 测试数据 验收 |
| **`提交材料/测试结果/` 可能为空** | README/部署与运行说明 索引了 `单元测试摘要.txt`、`基准运行摘要.json` 等 | 评审前执行 §5.5 生成；勿将 `输出样例.json` 当作实跑报告 |
| **示例 `输出样例.json` 为示意** | `_example: true` | 实跑结果写入 `output_real.json` 或 `测试结果/` |
| **工具调用粒度不一致** | DAG 为 `task_type`/mock handler；Review-Plus Agent 为 `agent-traces`；解析为 `parser_trace` | 对照 §2 映射表分项核查 |
| **全量 pytest 401** | 部分 HTTP 测试未统一注入 `API_TOKEN` | 使用 部署与运行说明 §4.1 推荐子集或带 Token 冒烟 |
| **历史 golden 数字口径** | 文档曾记载 9/9 通过、P95=134ms；与单次环境、测试数据 是否齐全强相关 | 以当次 `基准运行摘要.json` 为准 |
| **`storage/runs/traces/` 可能为空** | 未跑规划/DAG 则无文件 | 执行规划 `POST /api/v1/planning/execute` 或竞赛 `use_dag=true` 场景 |
